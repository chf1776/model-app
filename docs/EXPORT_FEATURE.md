# Export Feature Specification

> Revised 2026-03-19. Supersedes the original spec and the "Build Log Export" section in PROJECT_PROPOSAL.md. Key changes from previous revision: HTML export removed, live preview removed (generate-then-review workflow), gallery section moved before track sections, two cover styles (full-bleed + classic), typst Rust crate (library) instead of CLI.

---

## 1. Design Philosophy

The export is the trophy of the build. It's a shareable document — part personal archive, part showcase you'd send to friends or family. Photos are the star, layout makes it feel premium, and technical reference (paint formulas, materials) is appendix material for those who want it.

**Primary audience**: The builder themselves (personal archive) and friends/family. Secondary: other modelers and forum communities.

**Guiding principles**:
- Photos dominate every spread. Text supports photos, not the other way around.
- Track colors carry through from the app into the document as the primary accent system.
- The user curates what goes in. Smart defaults make the common case fast, but every section is togglable, every photo is selectable, and every narrative block is editable.
- "Quick Export" exists for people who just want the PDF now.
- Exports work at any stage of a build — the document adapts to show progress or completion naturally.
- No live preview. The user configures, exports, reviews the generated file, and re-exports if needed. This keeps the dialog simple and avoids the engineering cost of real-time Typst rendering.

---

## 2. Output Formats

Two export formats. The section curation (toggles, photo selections, narratives) applies to both — they render the same content in their respective formats.

### 2.1 PDF

The primary format. A polished, typeset document generated by the `typst` Rust crate (compiled in-process, no external binary). Professional typography, proper page breaks, headers/footers, and print-quality layout. See §4 for full page design specification.

### 2.2 ZIP

Archive for raw materials and long-term archival. Contains:
- `build-log.md` — Narrative Markdown file with the same structure as the PDF: project info, track sections with narrative text, photo references as relative Markdown image links, paint palette as a Markdown table, and build statistics.
- `images/` — Selected photos organized by track subfolder, plus a `gallery/` folder for gallery and hero photos.

```
{project-name}-export.zip
├── build-log.md
└── images/
    ├── gallery/
    │   ├── hero-shot.jpg
    │   └── port-side.jpg
    ├── hull-assembly/
    │   ├── step-4-dry-fit.jpg
    │   └── milestone.jpg
    └── superstructure/
        └── step-7-bridge.jpg
```

The Markdown file is designed to be readable in Obsidian, VS Code, or any Markdown viewer. Photo filenames are slugified from step/track names for readability.

## 3. Export Dialog

The export dialog is a full-screen modal opened from the Project Info card in the Overview zone (Export button) or from the completion flow prompt.

### 3.1 Layout

Two-panel layout (no live preview):

| Panel | Width | Content |
|-------|-------|---------|
| Left sidebar | 240px fixed | Section list with toggles, format selector, page size, page estimate |
| Main area | Flexible, scrollable | Section editor (changes based on selected section in sidebar) |

This matches the Settings page pattern already in the app.

### 3.2 Title Bar

Top bar spanning both panels:
- Left: "Export Build Log" title, project name + kit subtitle below
- Right: two buttons
  - **Quick Export** (secondary/outline): Generates a PDF immediately using current saved settings (or smart defaults if first export). Triggers system save dialog, then opens the file. No further interaction.
  - **Export** (primary, accent color): Triggers export in the selected format with current curation settings. For PDF: system save dialog → generate → open. For ZIP: system save dialog → generate → toast.

### 3.3 Left Sidebar: Section List

Ordered list of togglable sections. Each row shows:
- Section icon (monochrome, 16px)
- Section title (bold, 12px)
- Section subtitle (secondary text, 10px — brief data summary like "{N} photos" or "{N} colors")
- Toggle switch (on/off, clickable without selecting the section)

Clicking the row text/icon selects it (highlights with accent left border + subtle accent background tint) and loads its editor in the main area. Clicking the toggle enables/disables the section without selecting it.

**Section list** (default order):

| # | Title | Subtitle (dynamic) | Default |
|---|-------|---------------------|---------|
| 1 | Cover Page | Hero photo + project details | On |
| 2 | Gallery | {N} photos selected | On |
| 3 | Build Story | {N} tracks with photos & narratives | On |
| 4 | Paint Palette | {N} colors with swatches | On |
| 5 | Materials List | Kit + {N} accessories | Off |
| 6 | Assembly Map | Track structure diagram | Off |
| 7 | Build Statistics | Duration, steps, photos | On |

Note: Gallery appears before Build Story — the document leads with the finished model showcase, then shows the build process.

**Auto-disable rules**: Sections with no content are automatically disabled and shown as greyed-out in the sidebar. Specifically:
- Gallery: disabled if the project has zero photos of any type
- Build Story: disabled if the project has zero tracks
- Paint Palette: disabled if the project palette is empty
- Materials List: disabled if no accessories exist
- Assembly Map: disabled if the project has zero tracks
- Build Statistics: always available (shows zeroes at minimum)
- Cover Page: always available (falls back to text-only if no photos)

The user can still manually enable auto-disabled sections (they'll produce empty/minimal output).

Below the section list:

**Format selector**: Two radio buttons:
- **PDF** — selected by default.
- **ZIP** — Photos + Markdown archive.

**Page size** (PDF only, hidden when ZIP selected): Two radio buttons:
- **A4** (210×297mm) — selected by default.
- **US Letter** (8.5×11").

**Page/photo estimate**: Small info block showing:
- For PDF: "~{N} pages · {N} photos"
- For ZIP: "{N} photos · ~{size}MB"

Recalculates as sections are toggled and photos are selected/deselected.

### 3.4 Main Area: Section Editors

Each section has a dedicated editor view. The editor loads when the section is selected in the sidebar. If no section is selected, the main area shows a summary view (section checklist with photo/page counts).

#### Cover Page Editor

- **Cover style toggle**: Two buttons with small inline previews — "Full-bleed" and "Classic". Default: Full-bleed if a hero photo is set, Classic otherwise.
- **Hero photo selector**: Grid of candidate photos (all gallery, hero, and starred photos). Click to select one as the cover image. Selected photo shows accent border + checkmark badge. Default: the project's designated hero photo (the photo set via "Set as cover" in the gallery lightbox — stored as `hero_photo_path` on the project).
- **Title field**: Editable text input, pre-filled with project name.
- **Subtitle field**: Editable text input, pre-filled with "{manufacturer} {scale} · {kit_number}".

The cover page automatically pulls build date range, completion percentage (for in-progress builds), and track/step counts from the project data. These are not editable in the dialog.

#### Gallery Editor

Shows all project photos in a selectable grid. This is the full photo picker — the user decides exactly which photos appear in the Gallery section of the document.

Photo types included:
- Gallery photos (finished model shots uploaded to the Gallery card)
- Hero photo (the project's designated cover photo, stored as `hero_photo_path`)
- Starred progress photos
- Milestone photos
- Unstarred progress photos
- Reference photos (attached to steps)

**Filter tabs** above the grid:
- **All** ({total count})
- **Gallery** ({gallery + hero count})
- **Starred** ({starred count})
- **Progress** ({progress count})
- **Milestone** ({milestone count})
- **Reference** ({reference count})

Each photo thumbnail in the grid shows:
- Photo at natural aspect ratio, rounded corners (4px)
- Caption label at bottom (step name, track name, or user caption)
- Selection state: selected photos show accent border + small checkmark badge in top-right. Deselected photos show at 40% opacity.
- Type badge (small, top-left): "HERO" (warning yellow) for hero photo, "★" for starred, "REF" (muted) for reference photos.

Click to toggle selection. Selected count shown above the grid ("{N} of {total} selected").

**Default selection**: All gallery, hero, starred, and milestone photos selected. Unstarred progress and reference photos deselected. The user can override any individual selection.

#### Build Story Editor

One card per track, in display order. Each card shows:
- Track color dot (8px) + track name (bold, 12px)
- Step count + photo count for that track (secondary text, right-aligned)
- Narrative textarea: 2-3 lines tall, pre-populated with milestone notes if they exist, otherwise placeholder text ("Notes about {track name}..."). The user can write brief notes about technique, challenges, or tips. This narrative text appears in the PDF below the track header and above the photo grid.

Track cards are collapsible. Clicking the track header toggles the textarea visibility. Collapsed cards show a truncated preview of the narrative (first line) or "No notes" in tertiary text.

All progress and milestone photos for a track are included in its section automatically. Per-track photo selection is deferred — the Gallery editor handles global photo curation.

#### Paint Palette Editor

Read-only preview of what will appear in the document. Shows all paints from the project palette:
- Color swatch (20×20px rounded square)
- Paint code (monospace, secondary color)
- Paint name
- Brand (tertiary text, right-aligned)
- Purpose name if set (from palette entry)

All paints are included by default. Individual paint toggle (include/exclude) is deferred to v2. For v1, the palette is all-or-nothing via the section toggle.

#### Materials List Editor

Read-only preview. Shows:
- Kit info: name, manufacturer, scale, kit number
- Accessories list with type badges and owned/needed status

Section is all-or-nothing via the toggle. No per-item selection.

#### Assembly Map Editor

Read-only preview. Shows a description of what will be rendered:
- "Assembly map diagram showing {N} tracks and {N} steps"
- Small static preview of the map (a screenshot rendered from the existing React-based Assembly Map component in the Overview zone)

The actual diagram for the PDF is rendered during export as a static PNG by the frontend (offscreen render of the Assembly Map component), then passed to the Rust backend for embedding in the Typst document. See §8.5 for details.

#### Build Statistics Editor

Read-only preview of the stats that will appear:
- Build duration (days from first to last activity, or "In progress")
- Total steps / completed steps
- Total tracks / completed tracks
- Photo count (number selected for export)
- Paint count (palette entries)

## 4. PDF Page Design

The PDF is generated by the `typst` Rust crate (v0.12+) in the Rust backend. Typst templates define the layout; the backend populates them with structured project data and photo file paths.

### 4.1 Paper and Typography

- **Page size**: A4 (210×297mm) by default. Configurable to US Letter (8.5×11") in the export dialog sidebar.
- **Margins**: 20mm all sides. Cover page: 0mm for full-bleed, 20mm for classic.
- **Body font**: DM Sans, 11pt. Matches the app's UI typography. DM Sans is licensed under the SIL Open Font License — safe to embed in generated PDFs.
- **Heading font**: DM Sans, bold weights.
- **Color palette**: Warm cream paper background (#F8F6F3, matches app `--color-background`). Body text in near-black (#1A1A1A). Secondary text in medium grey (#666). Track accent colors from each track's assigned color. App accent (#4E7282, the default theme's "Kure Steel" teal-grey) used for global accents outside track-specific contexts (e.g., stat numbers, classic cover subtitle, decorative rules).

### 4.2 Cover Page — Full-Bleed Style

Full first page. No header, footer, or page number. Zero margins.

- **Full page**: Hero photo fills the entire page edge-to-edge. If the photo aspect ratio doesn't match the page size, it is center-cropped to fill.
- **Gradient overlay**: Bottom 35% of the page has a gradient from transparent to 80% black.
- **Title block** (overlaid on gradient, bottom-left, 20mm inset from edges):
  - Project name: 24pt bold, white, tight letter-spacing (-0.5).
  - Subtitle: 11pt semibold, white at 80% opacity. Format: "{MANUFACTURER} {SCALE} · {KIT_NUMBER}".
  - Metadata row: "Built {date range} · {N} steps across {M} tracks" — 8pt, white at 60% opacity.

**Fallback**: If no hero photo is selected for the cover, falls back to Classic style automatically.

### 4.3 Cover Page — Classic Style

Full first page. No header, footer, or page number. Standard 20mm margins.

- **Top 60%**: Hero photo (or box art if no hero photo exists), displayed large with subtle rounded corners (6px). Center-aligned. If neither hero photo nor box art exists, this area shows a solid rectangle in the app accent color (#4E7282) at 15% opacity.
- **Bottom 40%**: Title block on cream background.
  - Project name: 22pt bold, tight letter-spacing (-0.5), near-black.
  - Subtitle: 11pt semibold, app accent color (#4E7282), 0.5 letter-spacing. Format: "{MANUFACTURER} {SCALE} · {KIT_NUMBER}".
  - Metadata row: Three items side by side, each with an uppercase 7pt label (grey) and 9pt value (dark grey):
    - "Built" → date range (or "Started {date}" for in-progress builds)
    - "Steps" → "{completed}/{total} across {M} tracks"
    - "Progress" → "{N}%" (or "Complete" for finished builds)
  - Accent line: 40px wide, 2px tall, app accent color gradient fading to transparent. Decorative separator below metadata.

### 4.4 Gallery Pages

Dedicated section for the curated gallery photos selected in the export dialog. This section appears immediately after the cover — it's the finished model showcase.

**Section header** (top of first gallery page):
- "GALLERY" label (8pt uppercase, grey, 1.5 letter-spacing)
- "Finished Model" title (18pt bold, near-black). For in-progress builds: "Build Gallery" instead.

**Layout adapts to photo count:**

**5+ photos** (typical):
- Hero layout: First selected photo displayed large at 16:9, full page width, 6px rounded corners.
- Three-up row: Next three photos in equal columns at 4:3 aspect ratio, 8px gaps.
- Remaining photos: Continued in 2-column grid (8px gaps) on subsequent pages.

**1–4 photos**:
- All photos displayed large, one or two per page, at 16:9.

**0 selected photos**:
- Gallery section is omitted entirely from the document. The Gallery toggle in the sidebar auto-disables.

Each photo cell has a caption overlay at bottom: gradient from transparent to 40% black, white text 7pt with the photo's caption or source label (step name for progress photos, track name for milestones, user caption for gallery uploads).

**Reference photos**: If reference photos were selected in the Gallery editor, they appear in a separate subsection titled "Reference Images" after the gallery photos, using the same 2-column grid layout.

### 4.5 Track Section Pages

One section per track, in display order. Each track section in the PDF corresponds to a track card in the Build Story editor. A section may span one or more pages depending on photo count and narrative length.

**Section header** (top of first page for each track):
- Track color dot (12×12px rounded square) + "TRACK {N} OF {TOTAL}" label (8pt uppercase, grey, 1.5 letter-spacing)
- Track name: 18pt bold, near-black, -0.3 letter-spacing
- Completion indicator: "{completed}/{total} steps" or "Complete ✓" — 9pt, grey (or success green for complete)
- User narrative text: 9pt, medium grey, line-height 1.5, max-width 80% of page width. This is the text from the Build Story editor. Omitted if empty.

**Photo grid**: Below the narrative. 2-column grid with 8px gaps. Each photo cell:
- Photo at 16:10 aspect ratio, rounded corners (4px)
- Caption overlay at bottom: gradient from transparent to 50% black, white text 7pt

Photos are the progress and milestone photos for this track, in chronological order (by `created_at` timestamp).

**Milestone callout**: If the track has a milestone note (from the milestone dialog when all track steps were completed), a callout box appears after the photo grid:
- Left border: 3px solid, track color
- Background: track color at 7% opacity
- "◆ MILESTONE" label: 7pt bold uppercase, track color
- Milestone text: 9pt, dark grey

**Empty tracks**: Tracks with no photos and no narrative are omitted from the document. Tracks with a narrative but no photos show the header + narrative only.

### 4.6 Appendix: Paint Palette

Included when the Paint Palette section is enabled. Starts on a new page.

**Section header**: "APPENDIX" label (8pt uppercase grey) + "Paint Palette" title (18pt bold)

**Swatch strip**: Full-width horizontal bar showing all palette colors side by side, no gaps, rounded container (4px). Each color gets equal width. Provides an at-a-glance visual summary.

**Paint table**: Below the swatch strip. Bordered table with header row (grey background):
- Columns: Swatch (14×14 color square), Code (monospace), Name, Brand, Purpose
- Rows: One per paint in the palette, sorted by color family
- Purpose column shows the palette entry's purpose name if set (e.g., "Base coat", "Weathering wash")

### 4.7 Appendix: Build Statistics

Can appear standalone or combined with the Paint Palette on the same appendix page.

**Stats grid**: 3-column, 2-row grid of stat cards. Each card shows:
- Large number: 18pt bold, app accent color (#4E7282)
- Label: 7pt uppercase, grey, 1 letter-spacing

Stats displayed:
- **Days** — build duration (first activity to last, or to present for in-progress)
- **Steps** — completed / total
- **Tracks** — completed / total
- **Photos** — count included in this export
- **Paints** — palette entry count
- **Sessions** — distinct days with build log activity

### 4.8 Appendix: Materials List

When enabled, appears after the paint palette / stats on the appendix pages.

**Section title**: "Materials" (12pt bold)

**Kit info block**: Kit name, manufacturer, scale, kit number, Scalemates URL (if set, as clickable link in PDF).

**Accessories table**: Bordered table with columns: Name, Type (PE/Resin/Decal/Other), Brand, Status (owned/needed). Sorted by type, then alphabetically.

### 4.9 Appendix: Assembly Map

When enabled, renders as a full-width diagram on its own page.

The assembly map is rendered as a static PNG by the frontend using an offscreen render of the existing Assembly Map React component. The PNG is passed to the Rust backend and embedded in the Typst document. This reuses the existing layout and rendering code rather than reimplementing the diagram in Rust.

Track lines are horizontal with step nodes: filled circle for complete, open circle for pending, diamond for join event. Track colors match the app. Join point arrows connect sub-tracks to parent tracks. Replaced steps shown with strikethrough styling.

### 4.10 Running Headers and Footers

- **Header** (pages 2+): Project name, left-aligned, 8pt, grey. Current section name on right (e.g., track name on track section pages, "Gallery" on gallery pages, "Appendix" on appendix pages).
- **Footer** (pages 2+): Page number, right-aligned, 8pt, grey.
- **Cover page**: No header or footer.

### 4.11 Page Break Logic

Document order:
1. Cover page (always page 1)
2. Gallery section (new page)
3. Track sections (each track starts on a new page)
4. Appendix — paint palette + stats + materials (new page, combined where they fit)
5. Assembly map (new page)

Disabled sections are skipped entirely — no blank pages or placeholders.

Within a track section, photos are allowed to flow across pages. Typst handles widow/orphan control to avoid a single photo stranded on a new page.

### 4.12 In-Progress Build Adaptations

Exports are allowed at any stage of a build. The document adapts naturally:
- Cover metadata shows "Started {date}" instead of "Built {date range}", and includes a progress percentage.
- Gallery section title becomes "Build Gallery" instead of "Finished Model".
- Track sections show completion counts ("{completed}/{total} steps").
- Build Statistics show current progress figures.
- No watermark or special "WIP" treatment — the data speaks for itself.

## 5. ZIP Export

When the user selects ZIP format, the same section selections and photo curation apply. The backend generates the Markdown file and copies selected photos into the archive.

### 5.1 Markdown Structure

The `build-log.md` file follows this structure:

```markdown
# {Project Name}

{Manufacturer} {Scale} · {Kit Number}
Built: {date range or "Started {date}"}
{completed}/{total} steps · {N} tracks

---

## Gallery

![{caption}](images/gallery/{filename})
...

---

## {Track Name}

{narrative text}

![{caption}](images/{track-slug}/{filename})
...

> ◆ Milestone: {milestone text}

---

## Paint Palette

| Swatch | Code | Name | Brand | Purpose |
|--------|------|------|-------|---------|
| 🟫 | XF-64 | Red Brown | Tamiya | Base coat |
...

## Build Statistics

- Duration: {N} days
- Steps: {completed}/{total}
- Tracks: {completed}/{total}
- Photos: {N}
- Paints: {N}

## Materials

**Kit**: {name} — {manufacturer} {scale} ({kit_number})

| Name | Type | Brand | Status |
|------|------|-------|--------|
| ...  | PE   | ...   | Owned  |
```

Sections are omitted if disabled in the export dialog. Photo references use relative paths to the `images/` directory. The Assembly Map section is omitted from the ZIP Markdown (it's a visual diagram that doesn't translate to text).

### 5.2 Image Organization

Photos are copied from the stash to the archive with readable filenames:

```
images/
├── gallery/
│   ├── 01-hero-shot.jpg
│   └── 02-port-side.jpg
├── hull-assembly/
│   ├── 01-step-4-dry-fit.jpg
│   └── 02-milestone.jpg
└── superstructure/
    └── 01-step-7-bridge.jpg
```

Filenames are zero-padded index + slugified caption/step name. Original resolution is preserved. Subfolder names are slugified track names, with `gallery/` for gallery and hero photos.

## 6. Quick Export

The "Quick Export" button generates a PDF immediately using smart defaults, bypassing detailed curation:

- **All sections enabled** except Materials List and Assembly Map
- **Photo selection**: All gallery, hero, starred, and milestone photos. Unstarred progress and reference photos excluded.
- **No user narrative** — milestone notes used where available
- **Cover**: Full-bleed style with the project's designated hero photo. Falls back to Classic style with box art if no hero photo exists, or text-only Classic if neither exists.
- **Title**: Project name
- **Subtitle**: "{manufacturer} {scale} · {kit_number}"
- **Format**: PDF only
- **Page size**: A4

**Behavior**: Click Quick Export → system save dialog → generate PDF → open in default viewer → toast confirmation.

If the user has previously opened the export dialog and made curation changes, Quick Export uses those **saved settings** instead of the defaults above. This lets the user configure once, then Quick Export subsequently.

## 7. File Handling

### 7.1 PDF Export Flow

1. User clicks "Export" (or "Quick Export")
2. System save dialog opens with suggested filename: `{Project Name} Build Log.pdf`
   - Default location: user's Documents folder (or last-used export location if stored)
3. User picks location and confirms
   - If the user cancels the save dialog, nothing happens — the export dialog remains open (or, for Quick Export, nothing happens at all)
4. Export button shows a spinner + "Generating..." label. Dialog inputs become non-interactive during generation. For Quick Export, a non-blocking toast shows "Generating PDF..." with a spinner.
5. Rust backend generates the PDF
6. PDF opens automatically in the system default viewer (`open` command on macOS)
7. Toast notification: "Exported successfully · Show in Finder" (clicking "Show in Finder" reveals the file in Finder via `showItemInFolder`)
8. Export recorded in `export_history` table

**Error handling**: If PDF generation fails (e.g., missing photo file, Typst compilation error), the backend returns an error. The frontend shows an error toast: "Export failed: {reason}". The dialog remains open so the user can adjust and retry.

### 7.2 ZIP Export Flow

Same as PDF, but:
- Suggested filename: `{Project Name} Export.zip`
- ZIP does not auto-open (no default "viewer" for ZIPs)
- Toast: "Exported successfully · Show in Finder"

### 7.3 Export History

Each export is recorded in the `export_history` database table (already defined in V1 migration). The V1 schema includes 'html' in the format CHECK constraint — a new migration will update this to only allow 'pdf' and 'zip':

```sql
-- Current (V1):  CHECK(format IN ('html','pdf','zip'))
-- Migration VN:  Update CHECK to ('pdf','zip')

CREATE TABLE IF NOT EXISTS export_history (
  id          TEXT PRIMARY KEY,
  project_id  TEXT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  format      TEXT NOT NULL CHECK(format IN ('pdf','zip')),
  file_path   TEXT NOT NULL,
  created_at  INTEGER NOT NULL
);
```

The Project Info card in Overview shows the most recent export date and format. A "Show in Finder" link opens the containing directory. The file path is stored but not validated on read — if the file has been moved/deleted, "Show in Finder" silently does nothing.

## 8. Typst Integration

### 8.1 Dependency

The `typst` Rust crate is used as an in-process library. No external binary or sidecar.

```toml
# Cargo.toml
typst = "0.12"
```

Binary size impact: ~10–15MB added to the application binary. Runtime memory during compilation: ~30–50MB (transient, released after PDF generation). Generation speed: under 1 second for a typical 10–20 page document.

### 8.2 Template Architecture

Typst templates are bundled with the app as embedded resources (included via `include_str!` or Tauri's resource system).

```
src-tauri/resources/templates/
├── build-log.typ         # Main template (page setup, section routing)
├── cover-bleed.typ       # Full-bleed cover page layout
├── cover-classic.typ     # Classic cover page layout
├── gallery.typ           # Gallery section layout
├── track-spread.typ      # Track section layout
├── appendix.typ          # Paint palette, stats, materials
├── assembly-map.typ      # Assembly map embed (places the pre-rendered PNG)
└── fonts/
    └── DMSans-*.ttf      # Bundled font files (Variable, Bold, Italic — SIL OFL license)
```

The main template imports section modules and conditionally includes them based on section enable flags in the input data.

### 8.3 Data Flow

1. User clicks "Export" in dialog (or "Quick Export")
2. Frontend collects export configuration and sends it to the Rust backend via Tauri command:
   ```
   export_build_log {
     project_id: String,
     format: "pdf" | "zip",
     output_path: String,              // from save dialog
     page_size: "a4" | "letter",       // PDF only
     sections: SectionFlags,           // which sections are enabled
     selected_photo_ids: Vec<String>,  // photo IDs across all types (gallery, progress, milestone, reference)
     cover_config: CoverConfig,        // style ("bleed" | "classic"), hero_photo_id, title, subtitle
     track_narratives: HashMap<String, String>,  // track_id → narrative text
     assembly_map_png: Option<Vec<u8>>,          // pre-rendered PNG from frontend, if Assembly Map enabled
   }
   ```
3. Rust backend:
   a. Queries database for all project data (project, kit, tracks, steps, paints, accessories, photos, build log)
   b. Builds a structured data object matching the template's expected schema
   c. Resolves photo file paths from the stash directory
   d. For PDF: compiles Typst template with data and photo paths → PDF bytes → writes to `output_path`
   e. For ZIP: generates Markdown, copies photos into archive structure → writes ZIP to `output_path`
   f. Records export in `export_history` table
   g. Returns success + file path to frontend
4. Frontend opens the file (PDF only, via Tauri `shell.open`) and shows success toast

### 8.4 Typst World Implementation

The `typst` crate requires a "World" trait implementation that provides:
- **Source files**: The `.typ` template files (loaded from embedded resources)
- **Fonts**: DM Sans font files (bundled in resources)
- **Data**: Project data as a Typst dictionary (converted from the Rust struct)
- **Images**: Photo file paths resolved to the stash directory

This is implemented as an `ExportWorld` struct in `src-tauri/src/services/export.rs`.

### 8.5 Assembly Map Rendering

The assembly map is rendered by the frontend, not the Rust backend. The existing Assembly Map React component in the Overview zone already implements the full layout algorithm (track lines, step nodes, join arrows, colors). Rather than reimplementing this in Rust:

1. When the user enables the Assembly Map section and clicks Export, the frontend renders the Assembly Map component offscreen to a canvas element.
2. The canvas is exported as a PNG data URL, converted to bytes.
3. The PNG bytes are included in the export command payload (`assembly_map_png`).
4. The Typst template embeds the PNG as a full-width image on its own page.

This keeps the layout logic in one place and ensures the exported diagram matches what the user sees in the app.

## 9. Export Configuration Persistence

Export settings are stored per-project so the user's curation carries across sessions. When the export dialog opens, it loads the last-used settings. If no previous export config exists, smart defaults are used (see §6 for default values).

**Settings stored**:
- Section enable/disable flags (7 booleans)
- Selected photo IDs (array of photo IDs across all types)
- Cover style ("bleed" or "classic")
- Cover hero photo ID
- Cover title override (null = use project name)
- Cover subtitle override (null = use auto-generated)
- Track narrative text (per track_id → string map)
- Selected format ("pdf" or "zip")
- Page size ("a4" or "letter")

These are stored as a JSON blob in the `project_ui_state` table under the key `export_config`. This is the same table used for build zone UI state persistence (active step, zoom level, etc.).

**Stale data handling**: If persisted settings reference photo IDs or track IDs that no longer exist (e.g., photos deleted, tracks removed since last export), those IDs are silently dropped when loading. The dialog shows current state, not stale references.

---

## 10. Entry Points

The export can be triggered from two locations:

1. **Overview zone → Project Info card**: "Export" button in the card header or expanded view. Opens the full export dialog.
2. **Project completion flow**: When a project is marked complete, the completion confirmation includes an "Export your build?" prompt that opens the export dialog.

Both entry points open the same dialog with the same persisted settings.
